# 卡顿优化

## 概述

"卡顿"是用户最直观感受到的性能问题。程序界面停顿、操作无响应、游戏掉帧——这些都属于卡顿问题。

卡顿的本质是：**某些操作占用了过多的 CPU 时间，导致主线程无法及时完成一帧的工作**。

本模块从四个角度剖析常见卡顿原因：

| 示例文件 | 主题 | 常见症状 |
|----------|------|---------|
| `01_cache_miss.cpp` | CPU 缓存未命中 | 遍历数据时莫名其妙地慢 |
| `02_false_sharing.cpp` | 伪共享 | 多线程程序越加线程越慢 |
| `03_lock_contention.cpp` | 锁竞争 | 高并发下吞吐量骤降 |
| `04_blocking_io.cpp` | 阻塞 I/O | 操作文件时界面冻结 |

## 01 - CPU 缓存未命中

### 缓存层级

现代 CPU 有多级缓存，访问速度差异巨大：

| 层级 | 典型大小 | 延迟 | 说明 |
|------|---------|------|------|
| **寄存器** | ~1 KB | ~0.3 ns | 最快 |
| **L1 缓存** | 32-64 KB | ~1 ns | 每个核心独有 |
| **L2 缓存** | 256 KB-1 MB | ~3-5 ns | 每个核心独有 |
| **L3 缓存** | 4-64 MB | ~10-20 ns | 所有核心共享 |
| **主内存** | GB 级 | ~50-100 ns | 慢 50-100 倍 |

### 缓存行 (Cache Line)

CPU 不是按字节读取内存的，而是按**缓存行**（通常 64 字节）读取。这意味着：

- 访问一个 `int`，会把它附近的 64 字节都加载进缓存
- 如果下一次访问的数据就在这 64 字节内，就是**缓存命中**（快）
- 如果跳到很远的地方，就是**缓存未命中**（慢）

### 测试一：顺序 vs 随机访问

```
随机访问通常比顺序访问慢 3-10 倍
```

**原因**：顺序访问时，CPU 预取器能提前加载后续数据到缓存；随机访问时预取无效。

### 测试二：AoS vs SoA

**AoS (Array of Structs)** - 传统面向对象布局：

```cpp
// 每个粒子 48 字节，但更新位置只需要 x, vx (8 字节)
// 缓存利用率: 8/48 = 17%
struct Particle { float x, y, z, vx, vy, vz, r, g, b, a, life, size; };
std::vector<Particle> particles;
```

**SoA (Struct of Arrays)** - 面向数据布局：

```cpp
// 更新位置时，每个缓存行全是有用数据
// 缓存利用率: 100%
struct Particles {
    std::vector<float> x, y, z;
    std::vector<float> vx, vy, vz;
    // ...
};
```

当操作只涉及部分字段时，SoA 的缓存效率远高于 AoS。

### 测试三：矩阵遍历方向

C/C++ 的二维数组是**行主序**存储：

```
正确 (行优先遍历):     错误 (列优先遍历):
for (行)               for (列)
  for (列)               for (行)
    arr[行][列]            arr[行][列]
                           ↑ 每次跳跃 N*4 字节，大量 cache miss
```

### 优化建议

1. 优先使用 `std::vector`（连续内存），避免 `std::list`、`std::map`
2. 热数据紧凑排列，冷数据分离存放
3. 遍历顺序与内存布局一致
4. 大数据量批量处理时考虑 SoA 布局

## 02 - 伪共享 (False Sharing)

### 什么是伪共享

多个线程写入**不同的变量**，但这些变量恰好在**同一个缓存行**（64 字节）内：

```
缓存行 [counter_0 | counter_1 | counter_2 | counter_3 | ...]
         线程0写      线程1写      线程2写      线程3写
```

当线程 0 写入 `counter_0` 时，CPU 会使其他核心中包含这个缓存行的 L1 缓存失效。  
线程 1 要写入 `counter_1` 时，必须重新从 L2/L3 加载整个缓存行。  
结果：**每次写入都触发缓存同步，性能断崖式下降**。

### 修复方法：缓存行对齐

```cpp
// BAD: 紧密排列
struct { int64_t counter[8]; };  // 全在 1-2 个缓存行内

// GOOD: 每个计数器独占一个缓存行
struct alignas(64) PaddedCounter { int64_t value; };
PaddedCounter counters[8];  // 每个占独立的 64 字节
```

### 实际示例输出

```
False sharing (BAD)  : 200,000 us
Cache-aligned (GOOD) :  30,000 us
Speedup: 6.7x
```

线程越多，差距越大。

### 优化建议

1. 多线程写入的数据使用 `alignas(64)` 对齐
2. C++17 可用 `std::hardware_destructive_interference_size`
3. 使用线程局部变量累加，最后再合并

## 03 - 锁竞争

### 粗粒度锁 vs 细粒度锁

**粗粒度锁**（一把大锁保护所有数据）：

```cpp
class Cache {
    std::mutex mutex_;           // 所有操作共享一把锁
    std::unordered_map<int, int> data_;
public:
    void put(int k, int v) { lock_guard lk(mutex_); data_[k] = v; }
    int  get(int k)        { lock_guard lk(mutex_); return data_[k]; }
};
```

**分片锁**（按 key 分散到多把锁）：

```cpp
class ShardedCache {
    struct Shard { std::mutex mutex; std::unordered_map<int, int> data; };
    Shard shards_[16];
public:
    void put(int k, int v) {
        auto& s = shards_[k % 16];
        lock_guard lk(s.mutex);   // 不同 key 不竞争
        s.data[k] = v;
    }
};
```

### 读写锁 (shared_mutex)

读多写少的场景，多个读可以并行：

```cpp
class Counter {
    std::shared_mutex mutex_;
    int value_ = 0;
public:
    void increment() { std::unique_lock  lk(mutex_); value_++; }
    int  read()      { std::shared_lock  lk(mutex_); return value_; }
    //                 ^^^^^^^^^^^^^^^^^ 多个读线程可以同时进入
};
```

### 线程局部累加

**最有效的锁优化就是不加锁**：

```cpp
// BAD: 每次操作都加锁
for (int i = 0; i < N; ++i) { lock_guard lk(mutex); total += 1; }

// GOOD: 线程内部先累加，最后合并一次
int64_t localSum = 0;
for (int i = 0; i < N; ++i) { localSum += 1; }     // 无锁
total.fetch_add(localSum);                           // 只原子操作一次
```

差距可达 **100-1000 倍**。

## 04 - 阻塞 I/O

### 写入策略对比

| 策略 | 做法 | 性能 |
|------|------|------|
| 逐行 + flush | 每写一行调用 `flush()` | 最慢（每次都等磁盘） |
| 逐行（不 flush） | 写入后让系统决定何时刷盘 | 中等 |
| 批量写入 | 先拼接到内存，一次写出 | **最快** |

### 异步 I/O

文件读取不要阻塞主线程：

```cpp
// 同步 (阻塞主线程)
auto data = readFile("big_file.dat"); // 主线程卡住等 I/O

// 异步 (后台读取)
auto future = std::async(std::launch::async, []() {
    return readFile("big_file.dat");
});
// 主线程继续工作...
auto data = future.get(); // 需要时再取结果
```

### Windows 特有优化

| 方法 | 说明 |
|------|------|
| `FILE_FLAG_SEQUENTIAL_SCAN` | 告诉系统文件将被顺序读取，积极预读 |
| `FILE_FLAG_NO_BUFFERING` | 绕过系统缓存，减少内存拷贝 |
| 内存映射文件 (MapViewOfFile) | 延迟加载，按需读取，适合大文件 |
| Overlapped I/O | Windows 原生异步 I/O |
| I/O Completion Port (IOCP) | 高性能服务器标准方案 |

## 诊断思路

当遇到卡顿问题时，按以下顺序排查：

```
1. 确认是 CPU 卡顿还是 GPU 卡顿（任务管理器/GPU-Z）
   │
   ├─ CPU 高：
   │   ├─ 单线程 100%？→ 检查算法复杂度、缓存命中率
   │   ├─ 多线程但比单线程慢？→ 检查伪共享、锁竞争
   │   └─ CPU 不高但仍然卡？→ 检查 I/O 阻塞、等待
   │
   └─ GPU 高：
       ├─ Draw call 过多？→ 合批、实例化
       ├─ 像素填充率？→ 降分辨率测试
       └─ 显存不足？→ 纹理 LOD、压缩
```

## 总结

| 问题 | 症状 | 修复 |
|------|------|------|
| 缓存未命中 | 数据遍历慢 | 连续内存、SoA 布局 |
| 伪共享 | 多线程越多越慢 | `alignas(64)` 对齐 |
| 锁竞争 | 加线程无提升 | 分片锁、读写锁、局部累加 |
| 阻塞 I/O | 界面冻结 | 异步 I/O、批量操作 |
