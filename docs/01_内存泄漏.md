# 内存泄漏检测与修复

## 概述

内存泄漏是 C++ 程序中最常见的性能问题之一。程序分配了内存但未正确释放，导致内存占用持续增长，最终可能导致程序崩溃或系统变慢。

本模块包含 4 个由浅入深的示例，帮助你理解泄漏的产生原因和修复方法。

## 01 - 基本泄漏模式

### 常见的三种泄漏场景

**场景一：new 了不 delete**

```cpp
void leak() {
    int* p = new int(42);
    // 函数结束，p 被销毁，但它指向的内存永远无法释放
}
```

这是最基础的泄漏。指针变量 `p` 在栈上，函数返回时被销毁，但 `new` 分配的堆内存没有人来释放。

**场景二：异常路径泄漏**

```cpp
void process() {
    int* data = new int[1000];
    
    doSomething(); // 如果这里抛异常...
    
    delete[] data; // 这一行永远不会执行
}
```

即使你写了 `delete`，异常也会让控制流跳过它。这在大型项目中极其常见。

**场景三：容器中的裸指针**

```cpp
std::vector<Widget*> widgets;
widgets.push_back(new Widget());
widgets.clear(); // 只清空了指针，Widget 对象还在堆上!
```

`vector::clear()` 只销毁容器中的元素（指针值），不会调用 `delete`。

### 运行示例

```powershell
.\build\RelWithDebInfo\01_memory_leak_basic.exe
```

观察输出中的 Memory Report，它会显示每个泄漏的大小和源码位置。

## 02 - 智能指针修复

### 核心原则

> 现代 C++ 中几乎不应该直接使用 `new` / `delete`

### 三种智能指针

| 类型 | 所有权 | 适用场景 | 开销 |
|------|--------|---------|------|
| `unique_ptr` | 独占 | 单一所有者，生命周期明确 | **零开销** |
| `shared_ptr` | 共享 | 多个所有者需要共享同一对象 | 引用计数（原子操作） |
| `weak_ptr` | 观察 | 需要引用但不拥有 | 依赖 shared_ptr |

### unique_ptr 示例

```cpp
// 自动释放，异常安全
auto particle = std::make_unique<Particle>();
particle->update(0.016f);
// 离开作用域自动 delete，即使抛出异常也不会泄漏
```

### shared_ptr 在容器中

```cpp
std::vector<std::shared_ptr<Widget>> widgets;
widgets.push_back(std::make_shared<Widget>());
widgets.clear(); // 引用计数归零，Widget 自动释放
```

### 性能对比

示例程序会实际测量三者的性能差异：

- `unique_ptr` 对比裸指针：**几乎零开销**（编译器完全优化）
- `shared_ptr` 对比裸指针：有少量开销（原子引用计数），通常 5-15%

### 运行

```powershell
.\build\RelWithDebInfo\01_memory_leak_smart_ptr.exe
```

## 03 - 循环引用

### 问题描述

两个对象互相持有 `shared_ptr`，引用计数永远不为零：

```
A (ref=2) ──shared_ptr──> B (ref=2)
     ^                         │
     └────shared_ptr───────────┘
```

当外部的 `shared_ptr` 被销毁后：

```
A (ref=1) ──shared_ptr──> B (ref=1)
     ^                         │
     └────shared_ptr───────────┘
```

引用计数停在 1，永远不会到 0，内存永远不会被释放。

### 解决方案：weak_ptr

将"后向引用"改为 `weak_ptr`：

```cpp
struct Node {
    std::shared_ptr<Node> next;   // 前向：拥有
    std::weak_ptr<Node>   prev;   // 后向：观察（不增加引用计数）
};
```

### 实际场景：Observer 模式

事件系统持有监听者时应使用 `weak_ptr`：

```cpp
struct EventSystem {
    std::vector<std::weak_ptr<Listener>> listeners;
    
    void emit(const std::string& event) {
        for (auto it = listeners.begin(); it != listeners.end();) {
            if (auto sp = it->lock()) {
                sp->onEvent(event);
                ++it;
            } else {
                it = listeners.erase(it); // 监听者已死，自动清理
            }
        }
    }
};
```

### 运行

```powershell
.\build\RelWithDebInfo\01_memory_leak_circular_ref.exe
```

观察：BAD 版本的析构函数不会被调用，GOOD 版本会正确析构。

## 04 - Windows API 资源泄漏

### Windows 资源 ≠ C++ 内存

Windows 系统有自己的资源管理，智能指针无法直接管理：

| 资源类型 | 释放函数 | 常见场景 |
|----------|---------|---------|
| HANDLE | `CloseHandle()` | 文件、事件、互斥量、线程 |
| HBITMAP | `DeleteObject()` | GDI 位图 |
| HDC | `ReleaseDC()` / `DeleteDC()` | 设备上下文 |
| HKEY | `RegCloseKey()` | 注册表 |
| COM (IUnknown*) | `Release()` | COM 接口 |

### RAII 包装器

```cpp
class ScopedHandle {
    HANDLE handle_;
public:
    explicit ScopedHandle(HANDLE h) : handle_(h) {}
    ~ScopedHandle() {
        if (handle_ != INVALID_HANDLE_VALUE && handle_ != nullptr)
            CloseHandle(handle_);
    }
    HANDLE get() const { return handle_; }
    // 禁止拷贝，允许移动
};
```

### unique_ptr + 自定义删除器（更轻量）

```cpp
struct HandleDeleter {
    void operator()(HANDLE h) const {
        if (h && h != INVALID_HANDLE_VALUE) CloseHandle(h);
    }
};
using UniqueHandle = std::unique_ptr<void, HandleDeleter>;

// 使用
UniqueHandle hEvent(CreateEventW(nullptr, TRUE, FALSE, nullptr));
```

## 检测工具

### 编译期/运行时

| 工具 | 使用方式 |
|------|---------|
| **CRT Debug Heap** | `_CrtSetDbgFlag(_CRTDBG_ALLOC_MEM_DF \| _CRTDBG_LEAK_CHECK_DF)` |
| **Visual Studio 诊断工具** | Debug → Diagnostic Tools → Memory Usage |
| **本项目 MemoryTracker** | `perf::MemoryTracker::instance().enable()` + `report()` |

### 外部工具

| 工具 | 特点 |
|------|------|
| **Dr. Memory** | Windows 平台的 Valgrind 替代品 |
| **Application Verifier** | 微软官方检测工具，检测句柄/堆/锁的错误 |
| **LeakSanitizer** | Clang/GCC 内置 (MSVC 不支持) |
| **VMMap (Sysinternals)** | 可视化查看进程内存映射 |

## 总结

| 原则 | 说明 |
|------|------|
| 不用裸 new/delete | 用 `make_unique` / `make_shared` |
| 异常安全靠 RAII | 析构函数保证资源释放 |
| 后向引用用 weak_ptr | 打破循环引用 |
| Windows 资源也要 RAII | 自定义删除器或包装类 |
| 定期用工具检测 | CRT Debug、Visual Studio 诊断、VMMap |
