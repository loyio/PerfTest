# 帧率优化

## 概述

帧率优化是游戏和实时应用性能优化的核心议题。但"帧率优化"不仅仅是提高 FPS 数字——**帧步调（帧间隔的一致性）往往比帧率本身更重要**。

本模块覆盖三个关键主题：

| 示例文件 | 主题 | 核心问题 |
|----------|------|---------|
| `01_game_loop.cpp` | 游戏循环策略 | 物理模拟精度与帧率独立 |
| `02_frame_pacing.cpp` | 帧步调分析 | "60 FPS 但还是感觉卡" |
| `03_object_pooling.cpp` | 对象池 | 频繁分配导致的随机卡顿 |

## 01 - 游戏循环策略

### 三种主流方案

#### 方案一：可变时间步长

```cpp
while (running) {
    double dt = getElapsedTime();
    physics.update(dt);   // 直接用真实 dt
    render();
}
```

| 优点 | 缺点 |
|------|------|
| 实现简单 | 物理结果不可重现 |
| 不浪费 CPU | 极端 dt 导致穿模 |
| | 不同帧率下行为不同 |

#### 方案二：固定时间步长

```cpp
constexpr double FIXED_DT = 1.0 / 60.0; // 60Hz 物理
double accumulator = 0;

while (running) {
    double frameDt = getElapsedTime();
    accumulator += frameDt;
    
    while (accumulator >= FIXED_DT) {
        physics.update(FIXED_DT);    // 固定步长
        accumulator -= FIXED_DT;
    }
    render();  // 渲染时物理状态可能不是最新的
}
```

| 优点 | 缺点 |
|------|------|
| 物理确定性 | 渲染可能看到"旧"的物理状态 |
| 可重现结果 | 视觉上可能有微小抖动 |

#### 方案三：固定步长 + 渲染插值（最佳实践）

```cpp
while (running) {
    double frameDt = getElapsedTime();
    accumulator += frameDt;
    
    while (accumulator >= FIXED_DT) {
        previousState = currentState;
        physics.update(currentState, FIXED_DT);
        accumulator -= FIXED_DT;
    }
    
    double alpha = accumulator / FIXED_DT;
    // 在两次物理步之间插值，视觉上完美流畅
    RenderState renderState = lerp(previousState, currentState, alpha);
    render(renderState);
}
```

这是 **Unity、Unreal 等主流引擎** 采用的方案。

### 帧率限制器

精确的帧率限制需要 Sleep + spin-wait 组合：

```cpp
// Sleep 粗粒度休眠 (省 CPU)
sleep(剩余时间 - 2ms);

// Spin-wait 精确补齐 (精确到 us)
while (elapsed < targetFrameTime) { /* busy wait */ }
```

单独使用 `Sleep` 在 Windows 上精度只有 ~15ms（或 1ms with `timeBeginPeriod`），不够精确。


## 02 - 帧步调分析

### 为什么平均 FPS 会骗人

考虑两种场景，都是"平均 60 FPS"：

**场景 A：稳定 60 FPS**
```
帧时间: 16.6, 16.7, 16.6, 16.7, 16.6, 16.7 (ms)
→ 流畅
```

**场景 B：平均 60 FPS 但有卡顿**
```
帧时间: 10, 10, 10, 10, 10, 50, 10, 10, 10, 10, 10, 50 (ms)
→ 每隔几帧一次明显卡顿
```

两者平均帧率相同，但用户体验天差地别。

### 关键指标

| 指标 | 说明 | 重要性 |
|------|------|--------|
| 平均帧时间 | 所有帧的平均 | ★★☆ |
| P50 (中位数) | 50% 的帧低于此值 | ★★☆ |
| **P99** | 99% 的帧低于此值 | **★★★** |
| **Max** | 最慢的一帧 | **★★★** |
| Stddev | 帧时间标准差 | ★★★ |
| Jitter | 相邻帧时间差 | ★★★ |

> **P99 才是真正反映用户体验的指标。** 如果 P99 > 33ms，用户每 100 帧就会看到至少 1 次明显卡顿。

### 周期性卡顿的常见原因

| 现象 | 可能原因 |
|------|---------|
| 每 N 帧卡一次 | GC（垃圾回收）、定时任务 |
| 加载资源时卡 | 同步 I/O、shader 编译 |
| 场景切换卡 | 大量对象创建/销毁 |
| 随机卡顿 | 内存分配器碎片、系统中断 |

### Windows Sleep 精度问题

Windows 默认 `Sleep()` 精度约为 15.6ms（系统时钟分辨率），这意味着 `Sleep(1)` 实际可能睡 15ms。

**解决方案**：
- `timeBeginPeriod(1)` — 将系统时钟精度提高到 1ms
- `CreateWaitableTimerEx` + `HIGH_RESOLUTION` — Win10 1803+ 支持
- Sleep + spin-wait 混合策略

## 03 - 对象池

### 为什么频繁 new/delete 会导致卡顿

1. **系统调用开销**：`new` 最终调用操作系统的内存分配器，涉及内核态切换
2. **内存碎片化**：大量小对象分配/释放后，堆变得碎片化，后续分配变慢
3. **缓存不友好**：碎片化的堆导致对象分散在内存各处
4. **不确定性延迟**：有时分配器需要整理内存、向系统请求新页面

### 对象池原理

预先分配一大块内存，需要对象时从池中取出，用完后放回（而不是真正释放）：

```
初始状态:  [空][空][空][空][空][空][空][空]
acquire(): [使用][空][空][空][空][空][空][空]  → 返回第1个
acquire(): [使用][使用][空][空][空][空][空][空]  → 返回第2个
release(): [空][使用][空][空][空][空][空][空]  → 第1个归还
acquire(): [使用][使用][空][空][空][空][空][空]  → 复用第1个
```

### 性能差异

示例程序测量了三种场景的差异：

| 场景 | Heap new/delete | Object Pool | 提升 |
|------|----------------|-------------|------|
| 简单创建/销毁 | ~5000 us | ~500 us | **10x** |
| 粒子系统模拟 | ~15000 us | ~3000 us | **5x** |
| 碎片化后分配 | 不稳定 | 稳定 | **确定性** |

### 适用场景

| 场景 | 说明 |
|------|------|
| 粒子系统 | 每帧大量创建/销毁 |
| 子弹/投射物 | 游戏中最常见的短生命周期对象 |
| 网络消息包 | 固定大小的缓冲区复用 |
| UI 列表项 | 虚拟滚动列表中的行复用 |
| 线程池任务 | 避免频繁创建 std::thread |

## 帧率优化检查清单

### CPU 端

- [ ] 使用固定时间步长 + 渲染插值
- [ ] 热点数据使用连续内存 (vector)
- [ ] 频繁创建的对象使用对象池
- [ ] 将重工作分摊到多帧 (amortized work)
- [ ] I/O 在后台线程执行
- [ ] 减少锁竞争范围

### GPU 端（概念）

- [ ] 减少 Draw Call (合批、实例化)
- [ ] 使用 LOD (Level of Detail)
- [ ] 遮挡剔除 (Occlusion Culling)
- [ ] 着色器复杂度与分辨率平衡
- [ ] 避免每帧创建/销毁 GPU 资源

### 测量

- [ ] 不要只看平均 FPS，关注 P99 帧时间
- [ ] 使用 PresentMon / FrameView 分析帧步调
- [ ] 在 Release 模式 (最好 RelWithDebInfo) 下测试
- [ ] 测试最差场景 (大量物体、复杂场景)

## 总结

| 原则 | 详情 |
|------|------|
| 物理与渲染分离 | 固定步长物理 + 插值渲染 |
| 关注 P99 帧时间 | 比平均 FPS 更真实 |
| 避免动态分配 | 用对象池消除分配卡顿 |
| Sleep 精度很差 | 需要 spin-wait 补偿 |
| 分摊重工作 | 不要在一帧里做完所有事 |
