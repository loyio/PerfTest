# Windows 性能分析工具指南

## 概述

Windows 平台拥有丰富的性能分析工具生态。本文按类别整理了最常用的工具，涵盖从轻量级计时到硬件级分析的完整链路。

---

## 一、实时帧分析器

### 1.1 Tracy Profiler

> **推荐度：★★★★★** | 官网：https://github.com/wolfpld/tracy

Tracy 是目前游戏和实时应用领域**最受欢迎的开源 Profiler**，特点是极低开销、实时可视化。

**核心功能：**
- ✅ CPU Zone 分析（函数级耗时，支持嵌套）
- ✅ GPU 分析（支持 D3D11/D3D12/Vulkan/OpenGL）
- ✅ 帧时间可视化（帧时间直方图、帧率曲线）
- ✅ 内存分配追踪（每次 alloc/free 记录调用栈）
- ✅ 锁竞争分析（lock wait time、contention 可视化）
- ✅ 多线程时间线视图
- ✅ 远程 profiling（通过网络连接目标进程）
- ✅ 采集数据可保存为文件离线分析

**基本用法：**

```cpp
// 1. 引入头文件
#include <tracy/Tracy.hpp>

// 2. 标记帧边界
void MainLoop() {
    while (running) {
        FrameMark;              // 标记一帧
        Update();
        Render();
    }
}

// 3. 标记函数耗时
void Update() {
    ZoneScoped;                 // 自动记录此函数耗时
    PhysicsUpdate();
    AIUpdate();
}

// 4. 标记特定区域
void LoadAssets() {
    ZoneScopedN("Asset Loading");   // 自定义名称
    // ... 加载逻辑
}

// 5. 内存追踪
void* ptr = malloc(size);
TracAlloc(ptr, size);            // 记录分配
// ...
TracFree(ptr);                   // 记录释放
free(ptr);
```

**CMake 集成：**

```cmake
# 方法一：FetchContent
include(FetchContent)
FetchContent_Declare(tracy
    GIT_REPOSITORY https://github.com/wolfpld/tracy.git
    GIT_TAG v0.11.1
    GIT_SHALLOW TRUE
)
FetchContent_MakeAvailable(tracy)
target_link_libraries(your_target PUBLIC TracyClient)

# 方法二：vcpkg
# vcpkg install tracy
```

**使用流程：**
1. 代码中插入 Tracy 宏
2. 编译时定义 `TRACY_ENABLE`
3. 运行 Tracy Server（`tracy-profiler.exe`）
4. 启动你的程序 → Tracy Server 自动连接
5. 实时查看时间线、帧率、内存分配

**最佳实践：**
- Release 模式编译以获得真实性能数据
- 在关键路径添加 `ZoneScoped`，不需要给每个函数都加
- 使用 `ZoneColor(0xFF0000)` 标记异常区域
- 使用 `TracyMessageL("状态信息")` 在时间线上标注事件

### 1.2 Optick (Brofiler)

> **推荐度：★★★★☆** | 官网：https://github.com/bombomby/optick

Optick（原名 Brofiler）是一个轻量级 C++ 性能分析器，特别适合游戏引擎。

**特点：**
- 超低开销 (~0.2μs/event)
- 原生支持 D3D12 / Vulkan GPU 分析
- 内置帧分析功能
- 可视化 CPU/GPU 时间线
- 支持 Unreal Engine 插件

**基本用法：**

```cpp
#include <optick.h>

void GameLoop() {
    OPTICK_FRAME("MainThread");
    Update();
    Render();
}

void Update() {
    OPTICK_EVENT();
    // ...
}
```

### 1.3 Superluminal

> **推荐度：★★★★★** | 官网：https://superluminal.eu/

Superluminal 是一个**商业性能分析器**，在 AAA 游戏工作室中广泛使用。

**特点：**
- 采样 + Instrumentation 混合模式
- 极低开销的采样模式（不需要修改代码）
- 强大的调用树分析
- 支持 ETW 事件关联
- 自动检测资源竞争
- 无需重新编译即可分析（采样模式）

**适用场景：** 适合已有项目的性能分析，不需要插桩代码就能获得高质量分析结果。

## 二、CPU 深度分析器

### 2.1 Intel VTune Profiler

> **推荐度：★★★★★** | 官网：https://www.intel.com/content/www/us/en/developer/tools/oneapi/vtune-profiler.html

VTune 是 Intel 提供的**硬件级性能分析器**，能利用 CPU 硬件性能计数器 (PMU) 提供极深度的分析。

**核心分析模式：**

| 分析类型 | 说明 | 典型用途 |
|---------|------|---------|
| Hotspots | 采样分析 CPU 热点 | 找出最耗时的函数 |
| Microarchitecture | 微架构分析 | Cache miss、分支预测失败、流水线停顿 |
| Threading | 多线程分析 | 线程等待、负载不均、同步开销 |
| Memory Access | 内存访问分析 | NUMA 问题、缓存行为 |
| I/O | I/O 分析 | 磁盘/网络瓶颈 |

**命令行使用：**

```powershell
# 热点分析
vtune -collect hotspots -result-dir r001 -- your_app.exe

# 微架构分析（需要管理员权限）
vtune -collect uarch-exploration -result-dir r002 -- your_app.exe

# 多线程分析
vtune -collect threading -result-dir r003 -- your_app.exe

# 内存分析
vtune -collect memory-access -result-dir r004 -- your_app.exe
```

**VTune 能告诉你什么（Tracy 做不到的）：**

| 指标 | 说明 |
|------|------|
| CPI (Cycles Per Instruction) | 指令执行效率 |
| Cache Miss Rate | L1/L2/L3 各级缓存命中率 |
| Branch Misprediction | 分支预测错误率 |
| DTLB Miss | TLB 缓存未命中 |
| Memory Bandwidth | 内存带宽使用率 |
| Retiring 比例 | CPU 真正做有用功的比例 |

**典型分析流程：**
1. **Hotspots** → 找到最耗时函数
2. **Microarchitecture Exploration** → 分析该函数为何慢
3. 根据瓶颈类型优化（Cache miss → 改数据布局，Branch miss → 减少分支）

**注意事项：**
- VTune 在 AMD CPU 上也能工作（但硬件计数器支持有限）
- 微架构分析需要管理员权限
- 免费供个人和学术使用（Intel oneAPI 的一部分）

### 2.2 AMD μProf (uProf)

> **推荐度：★★★★☆** | 官网：https://developer.amd.com/amd-uprof/

AMD 平台的等价物，类似 VTune，但针对 AMD CPU 架构优化。

**特点：**
- 支持 AMD CPU 特有的硬件性能计数器
- IBS (Instruction Based Sampling) — AMD 独有的精确采样
- 电源分析 (Power Profiling)
- 支持 Linux 和 Windows

**使用场景：** 如果你主要在 AMD 平台开发，uProf 能提供更准确的微架构分析结果。

## 三、Windows 系统级工具

### 3.1 WPR / WPA (Windows Performance Recorder / Analyzer)

> **推荐度：★★★★★** | Windows SDK 自带

WPR/WPA 是微软官方的 **ETW (Event Tracing for Windows)** 分析工具套件，能看到操作系统级事件。

**WPR 录制：**

```powershell
# 启动录制
wpr -start CPU -start DiskIO -start FileIO

# 运行你的程序
.\your_app.exe

# 停止录制
wpr -stop output.etl

# 或使用自定义 profile
wpr -start myprofile.wprp -start GeneralProfile
wpr -stop output.etl
```

**WPA 中可分析的内容：**

| 图表 | 说明 |
|------|------|
| CPU Usage (Sampled) | CPU 采样热点 |
| CPU Usage (Precise) | 精确的上下文切换 |
| Disk I/O | 磁盘读写延迟和吞吐 |
| File I/O | 文件操作详情 |
| GPU | GPU 使用率 |
| Memory | 页错误、working set |

**何时使用 WPA 而非 Tracy/VTune：**
- 需要分析 **系统级交互**（驱动、内核、中断）
- 排查 **随机卡顿**（可能是其他进程抢 CPU 或磁盘）
- 分析 **DPC/ISR** 延迟（驱动和中断导致的延迟）
- 需要与其他系统事件 **关联**

---

### 3.2 PerfView

> **推荐度：★★★★☆** | 官网：https://github.com/microsoft/perfview

微软开源的 ETW 分析工具，特别擅长 **.NET 性能分析**，但也支持原生 C++ 通过 ETW。

**特点：**
- GC 分析（.NET 特有）
- 火焰图 (Flame Graph)
- 对比分析（两次运行的差异）
- 命令行和 GUI 双模式

---

### 3.3 PresentMon / FrameView / FCAT

> **推荐度：★★★★★** | 帧率分析专用

**PresentMon：**
- 微软开源工具，底层统计 Present 调用
- 能测量真正到显示器的帧时间（不是引擎内部的）
- 命令行工具

```powershell
# 录制特定进程的帧数据
PresentMon.exe -process_name your_app.exe -output_file frames.csv

# CSV 输出包含：
# TimeInSeconds, MsBetweenPresents, MsInPresentAPI, MsBetweenDisplayChange, ...
```

**FrameView：**
- NVIDIA 的 PresentMon 封装，带 GUI
- 自动生成统计报告（P50/P95/P99/1% Low/0.1% Low）
- 同时记录 GPU 功耗和频率

**用法对比：**
| 工具 | 用途 | 特点 |
|------|------|------|
| Tracy | 引擎内帧时间 | 细粒度，需插桩 |
| PresentMon | 实际显示帧时间 | 系统级，无需修改代码 |
| FrameView | 帧率报告 | 自动统计分析 |

---

### 3.4 GPUView

> **推荐度：★★★☆☆** | Windows SDK 自带

分析 GPU 调度的专用工具：
- 可视化 CPU 和 GPU 的工作队列
- 分析 Present 延迟链路
- 排查 GPU 瓶颈和调度问题

```powershell
# 录制
log.cmd  # GPUView 自带的录制脚本

gpuview.exe merged.etl
```

## 四、内存分析工具

### 4.1 Visual Studio 诊断工具

> **推荐度：★★★★☆** | Visual Studio 自带

**内存分析：**
- 诊断工具窗口 (Diagnostic Tools) → 内存使用率
- 拍摄堆快照 (Take Snapshot)
- 对比两个快照查看增长的对象

**有用快捷键：**
| 功能 | 快捷键 |
|------|--------|
| 启动调试并打开诊断工具 | F5 |
| 拍摄内存快照 | 诊断工具窗口中的相机图标 |
| 性能分析器 | Alt+F2 |

---

### 4.2 Dr. Memory

> **推荐度：★★★☆☆** | 官网：https://drmemory.org/

开源内存分析工具（类似 Linux 上的 Valgrind）：
- 检测内存泄漏
- 检测缓冲区溢出
- 检测使用未初始化内存
- 检测 use-after-free

```powershell
drmemory.exe -- your_app.exe
```

---

### 4.3 Application Verifier

> **推荐度：★★★★☆** | Windows SDK 自带

微软官方的运行时验证工具：
- 堆损坏检测
- 句柄泄漏检测
- 锁使用问题检测
- 线程问题检测

```powershell
# 启用对特定程序的验证
appverif.exe /verify your_app.exe

# 常用检测项
# - Heap: 堆损坏、缓冲区溢出
# - Handles: 无效句柄使用
# - Locks: 死锁检测
# - Memory: 内存问题
```

---

### 4.4 RAMMap / VMMap

> **推荐度：★★★★☆** | Sysinternals 工具

**RAMMap：** 系统级内存分析
- 物理内存使用分布
- 哪些文件占用了内存
- 系统缓存状态

**VMMap：** 进程级虚拟内存分析
- 虚拟地址空间布局
- 各类型内存的使用量（Heap、Stack、Image、Mapped File）
- 内存碎片可视化

---

### 4.5 Process Monitor / Process Explorer

> **推荐度：★★★★★** | Sysinternals 工具

**Process Monitor (ProcMon)：**
- 实时监控文件系统、注册表、网络、进程活动
- 排查"为什么程序启动慢"→ 看它在读什么文件
- 支持高级过滤和调用栈

**Process Explorer：**
- 增强版任务管理器
- 查看进程的句柄、DLL、线程
- CPU 使用率分解到线程级

## 七、实战工作流示例

### 场景：游戏出现随机卡顿

```
Step 1: 宏观定位
├── 用 Tracy 确认是 CPU 还是 GPU 瓶颈
├── 用 PresentMon 确认帧步调数据
└── 如果是系统级问题 → WPR/WPA

Step 2: CPU 卡顿
├── Tracy → 找到耗时最长的 Zone
├── VTune Hotspots → 确认热点函数
├── VTune Microarch → 分析为什么慢
│   ├── Cache Miss → 优化数据布局 (参考 02_卡顿优化)
│   ├── Branch Miss → 减少分支、使用 branchless
│   └── Memory Bound → 优化访问模式
└── Tracy 验证优化效果

Step 3: GPU 卡顿
├── Nsight/PIX → GPU Trace
├── 找到最耗时的 Pass
├── 分析 Shader 瓶颈
└── 优化 Draw Call 数量

Step 4: 内存卡顿
├── Tracy Memory → 确认是否有频繁分配
├── VTune Memory → 分析缓存利用率
└── 使用对象池替代频繁 new/delete
```

### 场景：优化启动时间

```
Step 1: Process Monitor → 观察启动时读了哪些文件
Step 2: WPR + WPA → 分析启动期间的 CPU 和 I/O 活动
Step 3: Tracy → 在初始化代码中插桩，找到最慢的模块
Step 4: 优化策略
├── 延迟加载非必要模块
├── 并行化独立的初始化步骤
└── 减少配置文件读取次数
```
